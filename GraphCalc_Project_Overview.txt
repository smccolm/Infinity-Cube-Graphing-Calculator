# GraphCalc Infinity Cube Project Overview

## 1. Purpose

GraphCalc is a native Windows 11 graphing calculator prototype for exploring three-dimensional functions inside a finite view of infinity. The app started as a rotatable 3D TAN Cube concept and has grown into an experimental visual math instrument with a front end, an internal API, a calculation backend, local result storage, test logging, level-of-detail controls, mesh rendering, and cinematic surface rendering.

The core idea is simple but powerful:

```text
Real X, Y, and Z can be infinite.
The screen cannot be infinite.
So the app maps real values into finite angular coordinates.
```

The display transform is:

```text
thetaX = atan(x)
thetaY = atan(y)
thetaZ = atan(z)
```

Because `atan(value)` approaches `-pi/2` and `pi/2`, every real value fits inside a finite cube:

```text
thetaX, thetaY, thetaZ in (-pi/2, pi/2)
```

The result is a finite cube where the boundary represents approach toward infinity. This is why the project can display functions that would normally fly off the page.

A better public name than TAN Cube is probably **Infinity Cube**. The technical description is:

```text
An arctangent-compactified 3D function viewer.
```

The practical name can be:

```text
Infinity Cube
```

The short explanation can be:

```text
Infinity Cube maps infinite X, Y, and Z into a finite rotatable cube using arctangent compactification.
```

## 2. What makes the project different

Normal 3D graphing tools usually show `z = f(x, y)` inside a finite rectangular domain. If a function grows very large, the tool either clips it, rescales it, or makes the graph unreadable.

GraphCalc does something different. It treats infinity as a visible boundary.

The app evaluates functions in real space:

```text
x = real X value
y = real Y value
z = f(x, y)
```

But it renders the result in compactified display space:

```text
rendered point = (atan(x), atan(y), atan(z))
```

That means a function can approach infinity without forcing the cube to grow or autoscale. The cube stays fixed.

This led to a design rule that became central to the project:

```text
For visible shape control inside the Infinity Cube, use atan(x) and atan(y), not only x and y.
```

Real-space radius:

```text
x*x + y*y
```

Projected visible radius:

```text
atan(x)*atan(x) + atan(y)*atan(y)
```

The second expression controls the shape as it appears in the compactified cube.

## 4. Core mathematical model

The user enters an expression using real-space variables:

```text
z = f(x, y)
```

The app samples compact projective coordinates, converts them to real values, evaluates the expression, then maps the output back to compact display space.

Sampling coordinate:

```text
p in (-1, 1)
theta = asin(p)
real = p / sqrt(1 - p*p)
```

This is equivalent to:

```text
real = tan(theta)
```

Then the app evaluates:

```text
realZ = f(realX, realY)
thetaZ = atan(realZ)
```

The rendered point is:

```text
(thetaX, thetaY, thetaZ)
```

All three axes are treated the same way. X, Y, and Z are each drawn in:

```text
[-pi/2, pi/2]
```

No autoscaling is allowed. Turning functions on and off must not change the cube scale.

## 5. BoundaryWeightedTanSampler

R8 introduced the sampler that became the accepted baseline. Earlier samplers were too center-heavy. They did not make the edge feel like infinity. The R8 sampler spends more samples near the `±pi/2` boundary while keeping enough detail in the center.

The sampler is:

```text
deterministic
function-invariant
LOD-controlled
edge-weighted
symmetric around zero
```

It does not adapt per function. Every function at the same LOD gets the same sample coordinates. That keeps comparison honest.

Core idea:

```text
medium detail in the center
higher detail near the infinity boundary
mandatory pi landmarks
mandatory near-boundary samples
```

Representative sampler code from `GraphCalc.Api/Program.cs`:

```csharp
static List<double> BuildBoundaryWeightedAxis(int sampleCount, double edgeInset, out int edgeCollarSamplesPerSide, out double tanThetaMax)
{
    // R8: deterministic boundary-weighted TAN sampler.
    // It keeps medium detail in the center and spends more samples near ±π/2,
    // where the finite TAN cube represents approach toward infinity.
    int n = Math.Max(9, sampleCount);
    if (n % 2 == 0) n++;

    double thetaMax = TanSpace.HalfPi - edgeInset;
    tanThetaMax = Math.Tan(thetaMax);
    const double gamma = 2.65;
    double denominator = n - 1.0;
    var thetaValues = new List<double>(n);

    for (int i = 0; i < n; i++)
    {
        double u = -1.0 + (2.0 * i / denominator);
        double sign = Math.Sign(u);
        double t = Math.Abs(u);
        double magnitude = thetaMax * (1.0 - Math.Pow(1.0 - t, gamma));
        thetaValues.Add(sign * magnitude);
    }

    thetaValues[n / 2] = 0.0;

    double[] landmarks =
    {
        -thetaMax, thetaMax,
        -31.0 * Math.PI / 64.0, 31.0 * Math.PI / 64.0,
        -15.0 * Math.PI / 32.0, 15.0 * Math.PI / 32.0,
        -7.0 * Math.PI / 16.0, 7.0 * Math.PI / 16.0,
        -3.0 * Math.PI / 8.0, 3.0 * Math.PI / 8.0,
        -Math.PI / 4.0, Math.PI / 4.0,
        -Math.PI / 8.0, Math.PI / 8.0,
        0.0
    };

    var used = new HashSet<int>();
    foreach (double target in landmarks)
    {
        if (target < -thetaMax || target > thetaMax) continue;
        int bestIndex = -1;
        double bestDistance = double.PositiveInfinity;
        for (int i = 0; i < thetaValues.Count; i++)
        {
            if (used.Contains(i)) continue;
            double distance = Math.Abs(thetaValues[i] - target);
            if (distance < bestDistance)
            {
                bestDistance = distance;
                bestIndex = i;
            }
        }
        if (bestIndex >= 0)
        {
            thetaValues[bestIndex] = target;
            used.Add(bestIndex);
        }
    }

    thetaValues.Sort();
    edgeCollarSamplesPerSide = thetaValues.Count(v => v >= 3.0 * Math.PI / 8.0);

    var values = new List<double>(thetaValues.Count);
    foreach (double theta in thetaValues)
    {
        values.Add(Math.Sin(theta));
    }
    return values;
}
```

The final returned values are projective `p` values. The API later converts them into display angles by using:

```csharp
double thetaX = Math.Asin(px);
double realX = ProjectiveToReal(px);
```

## 6. LOD model

The app has a Level of Detail menu:

```text
File > Level of Detail...
```

The purpose is to control sample density. The important point is that LOD changes the number of deterministic samples. It does not change the cube scale.

Example profiles:

```text
Draft       lower sample count
Standard    balanced working mode
High        dense mode
Ultra       heavy mode
Insane      very heavy mode
```

The accepted direction from R8 is:

```text
Standard and High should make edge behavior visible.
Ultra and Insane are available for heavy machines.
```

The user machine has very high headroom:

```text
NVIDIA RTX 4090
Ryzen 9 7950X
```

The current WPF renderer may become the limiting factor before the CPU or GPU does. Future Direct3D work would better use that hardware.

```csharp
private double ApplyFunction(string name, List<double> args)
{
    static void RequireCount(string name, List<double> args, int count)
    {
        if (args.Count != count)
        {
            throw new InvalidOperationException($"Function '{name}' expects {count} argument(s), but got {args.Count}.");
        }
    }

    return name switch
    {
        "sin" => One(name, args, Math.Sin),
        "cos" => One(name, args, Math.Cos),
        "tan" => One(name, args, Math.Tan),
        "atan" => One(name, args, Math.Atan),
        "asin" => One(name, args, Math.Asin),
        "acos" => One(name, args, Math.Acos),
        "sqrt" => One(name, args, Math.Sqrt),
        "abs" => One(name, args, Math.Abs),
        "log" => One(name, args, Math.Log),
        "log10" => One(name, args, Math.Log10),
        "exp" => One(name, args, Math.Exp),
        "floor" => One(name, args, Math.Floor),
        "ceil" => One(name, args, Math.Ceiling),
        "min" => Two(name, args, Math.Min),
        "max" => Two(name, args, Math.Max),
        _ => throw new InvalidOperationException($"Unsupported function '{name}'. Supported: sin, cos, tan, atan, asin, acos, sqrt, abs, log, log10, exp, floor, ceil, min, max.")
    };

    static double One(string name, List<double> args, Func<double, double> fn)
    {
        RequireCount(name, args, 1);
        return fn(args[0]);
    }

    static double Two(string name, List<double> args, Func<double, double, double> fn)
    {
        RequireCount(name, args, 2);
        return fn(args[0], args[1]);
    }
}
```

## 8. API calculation flow

The calculation API normalizes the request, builds the sampler, checks the cache, evaluates the function, then returns compactified points.

Representative code from `Program.cs`:

```csharp
app.MapPost("/calculate", (CalculationRequest request) =>
{
    string correlationId = Guid.NewGuid().ToString("N")[..8];
    request.Expression = request.Expression.Trim();
    request.SamplerName = TanSpace.SamplerName;
    request.AxisSampleCount = NormalizeAxisSampleCount(request.AxisSampleCount, maxGridSize);
    request.GridSize = request.AxisSampleCount;
    request.EdgeInset = NormalizeEdgeInset(request.EdgeInset);
    request.XMin = TanSpace.AxisMin;
    request.XMax = TanSpace.AxisMax;
    request.YMin = TanSpace.AxisMin;
    request.YMax = TanSpace.AxisMax;

    double thetaMax = TanSpace.HalfPi - request.EdgeInset;
    List<double> axisP = BuildBoundaryWeightedAxis(request.AxisSampleCount, request.EdgeInset, out int edgeCollarSamples, out double tanThetaMax);
    request.AxisSampleCount = axisP.Count;
    request.GridSize = axisP.Count;

    CalculationResultDto? cached = storage.TryGetCachedResult(request);
    if (cached != null)
    {
        cached.FunctionId = request.FunctionId;
        cached.Label = request.Label;
        cached.ColorHex = request.ColorHex;
        cached.IsVisible = request.IsVisible;
        return Results.Ok(cached);
    }

    var evaluator = new ExpressionEvaluator();
    var result = new CalculationResultDto
    {
        ResultId = Guid.NewGuid().ToString("N"),
        FunctionId = request.FunctionId,
        Label = request.Label,
        Expression = request.Expression,
        ExpressionHash = GraphStorage.ExpressionHash(request.Expression),
        ColorHex = request.ColorHex,
        IsVisible = request.IsVisible,
        SamplerName = request.SamplerName,
        LodName = request.LodName,
        AxisSampleCount = request.AxisSampleCount,
        EdgeInset = request.EdgeInset,
        GridSize = request.AxisSampleCount,
        XMin = TanSpace.AxisMin,
        XMax = TanSpace.AxisMax,
        YMin = TanSpace.AxisMin,
        YMax = TanSpace.AxisMax,
        Status = "Ready",
        FromCache = false,
        CreatedUtc = DateTime.UtcNow.ToString("O")
    };

    for (int ix = 0; ix < axisP.Count; ix++)
    {
        double px = axisP[ix];
        double thetaX = Math.Asin(px);
        double realX = ProjectiveToReal(px);

        for (int iy = 0; iy < axisP.Count; iy++)
        {
            double py = axisP[iy];
            double thetaY = Math.Asin(py);
            double realY = ProjectiveToReal(py);

            double realZ = evaluator.Evaluate(request.Expression, realX, realY, 0);
            double thetaZ = Math.Atan(realZ);

            result.Points.Add(new Point3Dto
            {
                X = thetaX,
                Y = thetaY,
                Z = thetaZ,
                Valid = true
            });
        }
    }

    storage.SaveResult(result);
    return Results.Ok(result);
});
```

The actual project includes error handling, invalid point handling, logging, and boundary behavior diagnostics.

## 9. Shared contracts

The shared project defines the API DTOs and the fixed TAN coordinate constants.

Representative code from `Contracts.cs`:

```csharp
public static class TanSpace
{
    public const string SamplerName = "BoundaryWeightedTanSampler";
    public const double HalfPi = Math.PI / 2.0;
    public const double AxisMin = -HalfPi;
    public const double AxisMax = HalfPi;
    public const double DefaultEdgeInset = 0.0001;

    public static double ToNormalized(double theta)
    {
        return Math.Clamp(theta / HalfPi, -1.0, 1.0);
    }

    public static string PrettyPi(double theta)
    {
        const double tolerance = 0.000001;
        if (Math.Abs(theta + HalfPi) < tolerance) return "-π/2";
        if (Math.Abs(theta + Math.PI / 4.0) < tolerance) return "-π/4";
        if (Math.Abs(theta) < tolerance) return "0";
        if (Math.Abs(theta - Math.PI / 4.0) < tolerance) return "π/4";
        if (Math.Abs(theta - HalfPi) < tolerance) return "π/2";
        return theta.ToString("0.###");
    }
}
```

The result points are stored as display-space angles:

```csharp
public sealed class Point3Dto
{
    // These values are TAN display angles, not raw real-space values.
    // X, Y, and Z are all in approximately [-π/2, π/2].
    public double X { get; set; }
    public double Y { get; set; }
    public double Z { get; set; }
    public bool Valid { get; set; } = true;
}
```

## 13. Storage and caching

The backend uses SQLite for local result storage.

The cache key includes the expression, sampler, LOD, and related parameters. This allows repeated rendering without recalculating.

Important behavior:

```text
Switching Mesh / Surface does not recalculate samples.
Switching Surface Quality does not recalculate samples.
Changing LOD recalculates checked non-empty functions.
Changing expression recalculates that function.
Duplicate expressions can share cached sample data while keeping separate function-row visibility and colors.
```

A prior issue was that cached results could exist without becoming active render geometry. That was fixed by making the render instance belong to the function row, not just the expression or result ID.

### 17.5 Radial squiggle of revolution


```text

large first lobe near center

ring-like surface in the Infinity Cube
```

This is one of the first formulas that felt like a new kind of visual math object. It is not just a normal `z = f(x,y)` graph. It is a radial function designed in compactified coordinates.

### 17.6 Cosmic horn / expansion funnel

```text
-(1/(0.000000001 + 0.35*sqrt(atan(x)*atan(x)+atan(y)*atan(y)) + 4*(atan(x)*atan(x)+atan(y)*atan(y)) + 180*(atan(x)*atan(x)+atan(y)*atan(y))*(atan(x)*atan(x)+atan(y)*atan(y))))
```

Behavior:

```text
negative Z horn
continuous taper
gradual expanding-universe style flare
point-like center region without a hard discontinuity
```

The projected radius is:

```text
q = atan(x)*atan(x)+atan(y)*atan(y)
```

The formula uses:

```text
sqrt(q)
q
q*q
```

Those terms shape the tip, body, and flare smoothly. Earlier two-term formulas produced a sudden handoff between the spike and the skirt. This single-denominator design is smoother.

### 17.7 Positive version of the cosmic horn


Behavior:

## 18. Designing functions for the Infinity Cube

The main lesson is that ordinary real-space intuition can be wrong in this viewer.

If you use:

```text
x*x + y*y
```

you are shaping the function in raw real-space radius.

If you use:

```text
atan(x)*atan(x) + atan(y)*atan(y)
```

you are shaping the function in visible compactified radius.

That distinction matters. A formula that looks skinny in ordinary coordinates can look like a broad mound in the Infinity Cube. A formula designed with projected radius can produce a much cleaner visual shape.

For radial objects, start with:

```text
q = atan(x)*atan(x)+atan(y)*atan(y)
```

Then shape the function with:

```text
sqrt(q)
q
q*q
sin(a*sqrt(q) + b*q)
exp(-c*sqrt(q))
```

Because the expression language does not support variables inside the formula editor, write `q` out manually.

## 19. Why the squiggle formula works

The radial squiggle formula is:

```text
(2.4*exp(-1.8*sqrt(q))*sin(12*sqrt(q) + 9*q))/(0.03 + sqrt(q))
```

where:

```text
q = atan(x)*atan(x)+atan(y)*atan(y)
```

In expanded calculator form:

```text
(2.4*exp(-1.8*sqrt(atan(x)*atan(x)+atan(y)*atan(y)))*sin(12*sqrt(atan(x)*atan(x)+atan(y)*atan(y)) + 9*(atan(x)*atan(x)+atan(y)*atan(y))))/(0.03 + sqrt(atan(x)*atan(x)+atan(y)*atan(y)))
```

The parts are:

```text
exp(-1.8*sqrt(q))     damping outward
sin(12*sqrt(q)+9*q)   oscillation with changing frequency
0.03+sqrt(q)          center control and radial amplitude shaping
```

When revolved around the Z axis, this creates rings and lobes that feel like a compactified ripple field.

## 20. Why the cosmic horn works

The cosmic horn formula is:

```text
-(1/(0.000000001 + 0.35*sqrt(q) + 4*q + 180*q*q))
```

where:

```text
q = atan(x)*atan(x)+atan(y)*atan(y)
```

Expanded calculator form:

```text
-(1/(0.000000001 + 0.35*sqrt(atan(x)*atan(x)+atan(y)*atan(y)) + 4*(atan(x)*atan(x)+atan(y)*atan(y)) + 180*(atan(x)*atan(x)+atan(y)*atan(y))*(atan(x)*atan(x)+atan(y)*atan(y))))
```

The parts are:

```text
0.000000001    keeps the center finite for calculation
0.35*sqrt(q)   sharpens the apparent point
4*q            controls the middle taper
180*q*q        controls the outer flare
negative sign  points the horn into negative Z
```

The important improvement was using one continuous denominator rather than adding a separate spike term and skirt term. The separate-term approach created an apparent sudden shoulder where the dominant term changed.

## 23. Current limitations

The current rendering is still WPF-based. That is useful for prototyping, but it is not the final high-performance path.

Known limits:

```text
many dense surfaces can become slow
surface sorting is not as strong as a GPU depth buffer
WPF is not fully using an RTX 4090
very dense edge behavior can still challenge the rasterizer
projection panels are simpler than the main view
```

The long-term rendering direction is Direct3D:

```text
GPU vertex buffers
GPU index buffers
proper depth buffer
shader-based lighting
per-pixel normals
hardware MSAA or TAA
instanced function rendering
```

```text

you are checking edge reach
```

Use Surface mode when:

```text
you want presentation quality
you want to understand broad shape
you want cinematic surfaces
you are exploring compactified forms visually
```

Use projected radius when designing shapes:

```text
atan(x)*atan(x)+atan(y)*atan(y)
```

Use real-space radius when you intentionally want raw-space behavior:

```text
x*x+y*y
```

## 26. Short glossary

```text
Infinity Cube
  A finite cube showing infinite real space through arctangent compactification.

TAN projection
  The practical sampling/rendering model where tan maps display samples to real values and atan maps results back into display space.

Arctangent compactification
  The mathematical idea of mapping an infinite real line into a finite interval with atan.

Projected radius
  A visible radius measured with atan(x) and atan(y).

Real-space radius
  Ordinary radius measured with x and y directly.

Mesh mode
  Technical wire rendering.

Surface mode
  Filled lit rendering.

LOD
  Level of Detail. Controls deterministic sample count and edge density.

BoundaryWeightedTanSampler
  The accepted sampler that gives medium center detail and more detail near the infinity boundary.
```
